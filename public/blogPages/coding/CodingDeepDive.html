MITTReads: a book-tracking app with data persistence

Background Summary

MITTReads was the second of three projects assigned in our SD-260: Intro to React course.  Given that this was the final course in the front-end portion of our program, this project had considerable complexity.  It demanded that we make use of all the disciplines that we’d learned thus far and placed great emphasis on our new knowledge of React hooks and React routers.  

Project Summary

The project itself had the following three primary components:

  • A server connected to a simple book database
  • A main page that displays the user’s books on “shelves” according to their status
  • A search page that allows the user to search for books in the database to add to their shelves

On the main page, the shelves visible to the user were “Currently Reading”, “Want to Read” and “Read”.  Each book displayed had a small pop-out menu button to select an alternate shelf or “none” to remove it entirely.  

On the search page, there was a search bar that would continuously filter results from the server as the user typed, controlled by debouncing (which will be explained in detail later).  

The server was managed by NPM JSON-server.  We started with an initial JSON file filled with over 400 books that would serve as our search database.

Top-level App

The app, at the top level, is surprisingly simple.  It makes heavy use of sub-components to delegate task complexity.  This part of the code displays our understanding of React-router by serving as the switch between our two user-viewable pages.  Here’s the relevant code:

import React from 'react';

import { BrowserRouter as Router, Switch, Route } from 'react-router-dom';

import AppBookList from './components/AppBookList';

import AppBookSearch from './components/AppBookSearch';

function App() {

  return (

    <Router>

      <div className="app">

        <Switch>

          <Route exact path="/">

            <AppBookList />

          </Route>

          <Route path="/search">

            <AppBookSearch />

          </Route>

        </Switch>

      </div>

    </Router>

  );

}

React-router is code intended to allow a single React application to span multiple web pages.  Here we see it in action.  The <Router> and <Switch> tags serve to allow switching behaviour, and the <Route> tags encapsulate separate pages.  Take special note of the path and exact path attributes: They appear in the user’s address bar.  Later in the code, we’ll have <Link> tags to connect them for navigation.  

Another point of note is the concept of “source of truth”.  When writing React apps, you use a concept called ‘state’ to determine what to show to your users.  If there are multiple different components that hold different states, you can run into problems.  It is advised to design your code around the idea that there’s only one source of truth.  In this project that source of truth was the server database, so there was no need to hold any states in this component.  

Main Page: AppBookList

The AppBookList component served as the main page for the user.  The code follows this simple plan:

  1. Pull data on all of the books from the server.  
  2. Sort those books into one of four categories: ‘currently reading’, ‘want to read’, ‘read’, and ‘none of the above’.  
  3. Display the first three categories as ‘shelves’ to the user.
  4. Allow the user to change the shelves of the books on display, via a dropdown menu.  
  1. If the user makes any such changes, update both the server database and the display.  
  1. Provide a navigation link to the search page.

Behold the relevant code:

  const [wantList, setWantList] = useState([]);

  const [readList, setReadList] = useState([]);

  const pullData = () => {

    getBooks().then(books => {

      const tempCurList = [];

      const tempWantList = [];

      const tempReadList = [];

      books.forEach((book) => {

        if (book.shelf) {

          switch (book.shelf) {

            case "wantToRead":

              tempWantList.push(book);

              break;

            case "read":

              tempReadList.push(book);

              break;

            case "currentlyReading":

              tempCurList.push(book);

              break;

          }

        }

      setWantList(tempWantList);

      setReadList(tempReadList);

    })

  }, []);

Here you can see our next lesson on display, React-hooks.  React-hooks allow us to move away from class-syntax code, back into functions: React-hooks give us access to two powerful tools: useState() and useEffect().  useState() replaces the older class-syntax of this.setState({}) with a much easier-to-read user-defined function.  useEffect() replaces all of the previous lifecycle methods componentDidMount(), componentWillUpdate(), and componentWillUnmount().  

Utility Functions

We cannot discuss the next section without first explaining some extra functions.  The first two serve as interactions with our server: getBooks() and changeBook().  They send simple fetch requests to either GET the book data or PUT a book onto a different shelf.  

Our third function, debounce(), is best described via a metaphor: Imagine a group of people go to a restaurant to order food and drink.  The server asks the first person for their order, and that person says “I’ll have a <food item>...” and the server immediately rushes off to place the order with the kitchen.  The server then returns, and the first person says “I also want <drink item>.”, and the server once again rushes off.  This pattern is inefficient and slow.  It would likely be better if the server were to take everyone’s full order, and then take all of the orders to the kitchen simultaneously.  ‘Debouncing’ modifies a function activation so that it doesn’t activate until after a short period of inactivity.  

We implement it thusly:

export const debounce = (func, delay) => {

  let inDebounce;

  return function() {

    const context = this;

    const args = arguments;

    clearTimeout(inDebounce);

    inDebounce = setTimeout(() => func.apply(context, args), delay);

  }

};

You call debounce() with two arguments: the function to be modified, and the timer delay after final activation.  debounce() essentially attaches a delay timer to the function, and repeated calls within that delay reset the timer.

Search Page: AppBookSearch

This component served as the user’s interface for adding additional books to their shelves.  The primary part of its complexity comes from the searching part itself:  The program would continuously filter search results as the user typed.  This was the reason for adding debouncing to our program:  We didn’t want to flood our system with a large number of erroneous results caused by rapid typing.  We debounced the search function behind a modest delay of 250ms.  

Here is the crucial part of the program code:

const filterFunction = debounce((str, list, cb) => {

  if (str) {  

    const reg = new RegExp(str, 'i'); // the 'i' flag ignores case

    const tempArr = list.filter((book) => {

      return reg.test(book.title);

    });

    cb(tempArr);

  } else {

    cb([]);

  }

}, 250);

Here you can see our function takes three arguments: The search string, the current list of books to search, and a callback function to indicate completion.  It converts the search string into a Regular Expression and activates the callback function with the newly-filtered list.
